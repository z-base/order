For each input `.txt` file, read the OCR-extracted receipt text and convert it into a corrected, normalized JSON-LD document using Schema.org.

The output must represent the receipt primarily as a `schema:Order`, but you must also use other Schema.org types and properties wherever they fit naturally and correctly, not only for the top-level order. For example, use Schema.org objects such as `Store`, `Organization`, `PostalAddress`, `Person`, `Offer`, `Product`, `QuantitativeValue`, `PropertyValue`, `PriceSpecification`, `Invoice`, `Service`, and other Schema.org types whenever they are the best semantic fit.

The output format must be valid JSON-LD. Use:

- `"@context": "https://schema.org"`
- either a single top-level object or `"@graph"` when multiple linked entities are needed

Your task is not just transcription. Your task is semantic normalization of noisy OCR receipt text into real-world structured data.

Critical correction rule:
You MUST NOT guess OCR corrections from intuition alone.
If merchant name, company number, address, product details, receipt number format, or other fields appear noisy or damaged, you may normalize them only when the correction is strongly supported by the receipt input itself.
Do not silently invent corrected values.
Only output corrected values when they are supported by direct receipt evidence.
If a value cannot be corrected with sufficient confidence from the input alone, preserve the most faithful observed value or omit the field rather than hallucinating.

Critical unresolved-field rule:
If a field is clearly present on the receipt but its value is unreadable or not recoverable with sufficient confidence, represent that field as unresolved instead of guessing.
Use an explicit empty string as the machine-readable unresolved marker only when the field label or field semantics are themselves clear.
For unresolved non-core facts, prefer a `PropertyValue` with a normalized `name` and `"value": ""`.
This allows reviewers and downstream software to detect that the field exists on the receipt but still needs human confirmation.
If the field label or semantics are also unclear, omit the field rather than inventing a synthetic label.

Sensitive-data rule:
For personal, sensitive, or reviewer-relevant fields, be stricter, not looser.
If a name, account identifier, member identifier, personal reference, or similar sensitive field is noisy and cannot be reconstructed with high confidence from the receipt alone, do not guess it.
Prefer an unresolved empty value when the field is clearly labeled, or omission when even the field identity is unclear.

Critical evidence-boundary rule:
You MUST NOT add factual information that is absent from the receipt.
External facts are not part of the supervision target.
If a merchant website, canonical branch name, alternate company number, expanded address component, URL, or other fact is not present on the receipt, do not add it unless the receipt itself explicitly points to it.
Do not add:

- merchant URLs unless printed on the receipt
- alternate registration numbers derived from a known company record
- expanded branch names not shown on the receipt
- inferred locality/country/postal detail that is not actually printed
- normalized product details beyond what the receipt supports

Validation rule:
When a receipt value is noisy but still plausibly recoverable, you SHOULD validate the guess with web search before committing to the correction.
Web search is for validation of receipt-supported guesses, not for enrichment.
If the receipt does not provide enough evidence to form a grounded candidate value, do not use web search to invent one.

Grounding rule:
Every important correction must be grounded in the receipt input itself.
Web validation may confirm a receipt-supported candidate reading, but it MUST NOT introduce new facts the receipt never contained.
Use only the receipt evidence plus constrained validation of receipt-supported guesses.

Grounding boundary:
Grounding can justify choosing between plausible readings of receipt evidence.
Web validation can help confirm one of those plausible readings.
Neither grounding nor web validation justifies adding extra fields or values that the receipt never contained.

Schema rule:
Use Schema.org objects wherever they fit, not just plain strings.
Examples:

- seller as `Store`, `Organization`, or `LocalBusiness`
- address as `PostalAddress`
- line items as `Offer`
- purchased goods as `Product`
- quantities as `QuantitativeValue`
- ad hoc extracted facts as `PropertyValue`
- totals as `PriceSpecification` when appropriate
  Choose the most semantically correct Schema.org structure possible.

Output quality rules:

- Output only JSON-LD
- Output valid JSON
- Do not include markdown fences
- Do not include explanations
- Do not include commentary
- Do not include citations inside the JSON-LD
- Do not include raw OCR blobs unless explicitly required
- Do not include provenance/debug/audit artifacts unless explicitly required
- Prefer corrected canonical values over noisy OCR fragments when supported by the receipt
- Preserve uncertainty conservatively: do not overclaim
- Prefer omission over enrichment when a fact is not directly evidenced by the receipt
- Weak receipts are allowed; the output may be sparse if the receipt is sparse
- Do not use fake placeholder values such as "Unknown item" or "Item on receipt"
- Do not convert unclear numeric fragments into product names, people, addresses, or other specific facts without strong receipt evidence
- Use empty-string unresolved markers only for clearly identified fields, not as a substitute for inventing structure

Receipt interpretation rules:

- Treat the receipt as an `Order` by default unless another Schema.org type is clearly more appropriate in addition
- Use `paymentStatus: "https://schema.org/PaymentComplete"` when the receipt clearly shows completed payment
- Use the receipt merchant as the seller
- Represent each purchased line item as an `Offer` linked to a `Product` or `Service`
- Include order number / document number when present
- Include order date in ISO 8601 format when it can be determined reliably
- Include `priceCurrency`
- Include totals, discounts, rounding adjustments, tendered cash, and change where present
- Include cashier, member number, notices, and other leftover receipt facts as `additionalProperty` using `PropertyValue` when no better Schema.org property exists

Conservatism rules:

- If a customer name appears on the receipt but is not explicitly labeled, do not assert it as customer unless the evidence is strong
- If a field is ambiguous, prefer omission over fabrication
- If multiple corrections are possible, choose the one best supported by the receipt evidence
- Do not create fake IDs, fake addresses, fake companies, fake products, or fake interpretations
- If a field is clearly present but the value is not recoverable, keep the field unresolved with an empty string only when the field identity is clear
- If both the label and the value are unclear, omit the field rather than inventing a label just to preserve structure

Normalization rules:

- Normalize dates to ISO 8601
- Normalize currency codes where possible, e.g. `MYR`, `EUR`, `USD`
- Normalize numbers as clean machine-readable strings or numbers as appropriate
- Normalize merchant names when supported by the receipt
- Normalize addresses into structured postal address fields when supported by the receipt
- Normalize GTIN / barcode values when supported by the receipt

Reference material for structure:

- https://schema.org/PaymentComplete
- https://www.w3.org/TR/json-ld11/
- https://json-ld.org/spec/latest/
